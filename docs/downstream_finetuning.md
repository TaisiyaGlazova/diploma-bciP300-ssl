# Stage 5 — Downstream Fine-Tuning and Evaluation

## Overview

На данном этапе проводится downstream-оценка SSL-предобученного энкодера в задаче бинарной классификации P300:

**Target vs Non-Target**

Цель эксперимента — проверить гипотезу:

> Self-supervised предобучение позволяет сократить объём индивидуальной калибровки.

Сравниваются:

* **Scratch baseline** — обучение модели с нуля
* **SSL + Fine-tuning** — использование предобученного encoder

---

## Model

Downstream-модель состоит из:

* **Encoder** — UNet1D (предобученный на SSL)
* **Head** — Global Average Pooling → Linear(512 → 2)

```text
(B, 14, 208) → encoder → (B, 512, 13)
→ GAP → (B, 512)
→ Linear → (B, 2)
```

Используется минимальная классификационная голова, чтобы оценивать вклад представлений encoder.

---

### BCI Competition III Dataset II (дополнительная проверка)

## Experimental Protocol

### BigP3BCI (основной протокол)

Используется межсубъектное разбиение.

#### Dev-настройка

Этапы:

1. Быстрый предварительный подбор:

   * `subj_054`, `subj_065`, `subj_090`, `subj_094`
   * (выбраны из benchmark-части)

2. Финальная Dev-оценка:

   * проводится на выделенной dev-подвыборке

#### Test-оценка

* проводится на оставшихся benchmark-субъектах
* субъекты, использованные на Dev, исключаются

👉 Таким образом:

* исключается утечка данных
* обеспечивается честная межсубъектная оценка

---

### BCI Competition III

Используется классический протокол:

* **subjA** — Dev
* **subjB** — Test

---

## Calibration Protocol

Для каждого субъекта:

* 70% → `calibration_pool`
* 30% → `test_rest`

Формируются вложенные подвыборки:

```text
p ∈ {10, 20, 40, 60, 100}
calib_10 ⊂ calib_20 ⊂ ... ⊂ calib_100
```

Это моделирует различные объёмы калибровочных данных.

---

## Normalization

* отдельно для каждого субъекта
* отдельно для каждого уровня p
* рассчитывается только по `Calib_p`

Применяется к:

```text
Calib_p + Test_rest
```

Без утечек данных.

---

## Training

* Loss: **Weighted CrossEntropy** (учёт дисбаланса 1:5)
* Optimizer: **AdamW**

---

## Fine-Tuning Strategies

Рассматриваются четыре стратегии:

* **Full Fine-Tuning (full_ft)**
  Полное дообучение encoder и head

* **Low LR Encoder (low_lr_encoder)**
  Замедленное обучение encoder

* **Partial Fine-Tuning (partial_ft)**
  Обучается только верхний блок encoder

* **Warmup Fine-Tuning (warmup)**
  Двухэтапное обучение:

  * сначала head
  * затем encoder + head

Все стратегии анализируются и сравниваются.

---

## Dev-Tuning Results

Основные наблюдения:

* **Partial fine-tuning** показывает наилучший ROC-AUC при больших p
* **Warmup** демонстрирует более стабильное поведение
* **Low LR encoder** показывает худшие и нестабильные результаты

---

## Hyperparameter Tuning

Рассматривались:

* `lr_encoder ∈ {1e-5, 3e-5}`
* `lr_head ∈ {1e-4, 3e-4}`
* `weight_decay ∈ {1e-4, 1e-3}`

### Итоги:

* `lr_encoder = 3e-5` — лучше
* `lr_head = 3e-4` — лучше
* `weight_decay = 1e-3` — стабильнее

---

## Final Protocol

Используется единый набор гиперпараметров:

```text
lr_encoder = 3e-5
lr_head = 3e-4
weight_decay = 1e-3
```

---

## Metrics

* Основная: **ROC-AUC**
* Дополнительная: **F1-score**

---

## Results

### 1. SSL vs Scratch (одинаковый p)

* при p = 10–40:

  * наблюдается улучшение ROC-AUC (partial_ft, warmup)
  * различия незначимы

* при p ≥ 60:

  * **full_ft показывает лучшие результаты**

* при p = 100:

  * full_ft даёт статистически значимое улучшение

---

### 2. Сравнение с scratch(p = 100)

* при p ≤ 40:

  * SSL хуже baseline

* при увеличении p:

  * разрыв уменьшается

* при p = 60:

  * full_ft ≈ scratch(100)

* при p = 100:

  * full_ft > scratch(100)

---

### 3. Снижение объёма калибровки

Ключевой результат:

> SSL позволяет сократить объём калибровки примерно на 30%

* full_ft при p = 40 ≈ scratch при p = 60
* различия незначимы
* ~62% субъектов не теряют качество

---

### 4. Общие выводы

* SSL не заменяет полностью калибровку
* при малых p эффект ограничен
* при достаточном p:

  * улучшает качество
  * повышает эффективность использования данных

---

## Additional Observations

* высокая межсубъектная вариабельность
* partial_ft и warmup стабильнее при малых p
* low_lr_encoder — худшая стратегия

---

## Conclusion

SSL-предобучение:

* улучшает качество моделей при достаточном объёме данных
* позволяет снизить объём калибровки (~30%)
* но не устраняет необходимость индивидуальной настройки полностью

Данный этап подтверждает пользу SSL для задачи P300 BCI при условии достаточного объёма калибровочных данных.
